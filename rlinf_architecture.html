1. 项目概述

是什么

RLinf（Reinforcement Learning Infrastructure / Infinite） 是一个基于 Ray 的开源分布式 RL 训练框架， 同时支持 具身 AI（机械臂、人形机器人）和 智能体/推理 AI（LLM 推理、代码生成、搜索）。

相比现有框架，最高可获得 2.434× 吞吐量提升。
核心设计理念
① Worker 解耦：Rollout、Actor、Reward、Critic 各自独立的 Ray Actor，互不干扰
② Channel 通信：Worker 间通过 FIFO Channel 异步传递数据，无需直接调用
③ 混合后端：推理用 vLLM/SGLang，训练用 FSDP/Megatron，可任意组合
④ Hydra 配置：所有组件通过 YAML 配置，支持层级覆盖，无需改代码
支持场景
具身 AI
ManiSkill · LIBERO · IsaacLab · CALVIN · MetaWorld · RoboCasa · RoboTwin · Habitat · FrankaSim
真实机器人：Franka 臂 · XSquare Turtle2 · Robotiq 夹爪
推理/智能体
数学推理 · 代码生成 · 网络搜索 (SearchR1) · 多智能体 (WideSeek-R1)
监督微调
VLA 微调 (Qwen2.5-VL/Qwen3-VL)
支持的 RL 算法
On-Policy
PPO · GRPO · DAPO · Reinforce++
Off-Policy
SAC · CrossQ · RLPD · SAC-Flow · DSRL
支持的模型
具身 VLA
OpenVLA · OpenPI (π₀/π₀.₅) · GR00T · LingBot-VLA · Dexbotic · CNN/MLP/Flow
语言推理
Qwen2.5 · Qwen3 · Qwen3-MoE · DeepSeek-R1
视觉语言
Qwen2.5-VL · Qwen3-VL

3. 推理/智能体 RL 训练流（ReasoningRunner）

关键更正： 权重同步（sync_weights）发生在每步的最开始（rollout 之前），而不是训练结束后。 这意味着每步 rollout 使用的是上一步训练完成后同步过来的最新策略权重。 源码：reasoning_runner.py:450-451

4. 具身 RL 训练流（EmbodiedRunner）

与 ReasoningRunner 的核心区别： 具身场景引入了 Env Worker 与 Rollout Worker 的双向交互循环； 优势函数（GAE advantages）在 Actor Worker 内部计算，不需要独立 Critic Inference； 权重同步按 weight_sync_interval 间隔触发，不是每步都同步。 源码：embodied_runner.py:251-298

6. 部署模式（Placement Strategies）

核心配置： cluster.component_placement 决定哪些组件部署在哪些节点/GPU 上。 格式为 "worker1,worker2: node_spec"，all 表示所有节点。
模式① Collocated
actor,rollout,reward: all
节点 0 · GPU 0-7（共享）
Actor Worker（FSDP 训练）
Rollout Worker（vLLM 推理）
Reward Worker
优点：配置简单，省内存
缺点：训练和推理争抢 GPU
适用：单节点 / 快速验证
模式② Disaggregated
actor: node0 · rollout: node1
节点 0 · GPU 0-3（训练）
Actor Worker（FSDP 训练）
sync_weights Bucket ~128MB
节点 1 · GPU 0-7（推理）
Rollout Worker（vLLM）
Inference Worker（logprobs）
优点：训练/推理互不抢占 GPU
性能：最高 2.434× 吞吐提升
适用：多节点生产训练
模式③ AUTO
component_placement: AUTO
动态调度器
SchedulerWorker
在线探测最优 Placement
优点：自适应，无需手动调参
工具：run_placement_autotune.sh
适用：不确定最优配置时
单节点配置示例
# 所有组件共享一台机器所有 GPU
cluster:
  num_nodes: 1
  component_placement:
    actor,rollout,reward: all

# 具身场景同时包含 env worker
cluster:
  num_nodes: 1
  component_placement:
    actor,env,rollout: all
多节点分离配置示例
# 16 节点：训练和推理分离
cluster:
  num_nodes: 16
  component_placement:
    actor,rollout,reward: all

# 或精确指定节点
cluster:
  num_nodes: 4
  component_placement:
    actor: 0-1     # 节点 0-1 训练
    rollout: 2-3   # 节点 2-3 推理

8. 配置文件详解（YAML）

配置系统：使用 Hydra，支持 defaults 组合和命令行覆盖。 配置分为 7 大顶层字段，分别对应系统各层。
cluster — 集群配置
num_nodes节点数量（训练集群规模）
component_placement组件 → GPU 节点分配策略
runner — 训练控制
task_typereasoning / embodied / sft
max_epochs / max_steps最大训练轮数/步数
save_interval每多少步保存 checkpoint
resume_dir断点续训目录（null / "auto" / 路径）
weight_sync_interval权重同步间隔（具身专用）
logger.logger_backends["tensorboard"] / ["wandb"] / ["swanlab"]
algorithm — RL 算法超参
adv_typegrpo / gae（PPO 用 gae）
group_size每条 prompt 生成的样本数（GRPO: 8-16）
kl_betaKL 惩罚系数（0 = 不加 KL）
recompute_logprobs是否重计算 logprobs（PPO/GRPO+KL 需要 True）
clip_ratio_high/lowPPO clip 上/下界（通常 0.2）
n_minibatches每步拆分的 minibatch 数
normalize_advantages是否归一化优势（推荐 True）
sampling_params.temperature生成温度（训练/评估可分别设置）
rollout — 推理引擎
generation_backendvllm / sglang / huggingface
model.model_path推理模型路径（HuggingFace 格式）
model.precisionbfloat16 / float16 / float32
recompute_logprobs具身场景的 logprob 重计算开关
actor — 训练后端
training_backendfsdp（小模型） / megatron（大模型）
model.model_path训练模型路径
micro_batch_size梯度累积的最小 batch
global_batch_size等效全局 batch size
optim.lr学习率（推荐 1e-6 ~ 1e-5）
fsdp_config.sharding_strategyfull_shard / hybrid_shard / no_shard
reward — 奖励模型
use_reward_modelfalse（规则奖励） / true（学习型奖励模型）
reward_typemath / code / vqa / chunk_level 等
reward_scale奖励缩放系数
critic — 价值函数（仅 PPO）
use_critic_modelfalse（GRPO 不用）/ true（PPO 必须）
training_backendfsdp / megatron
env — 具身环境（具身专用）
train.total_num_envs并行训练环境数量
train.max_episode_steps单 episode 最大步数（防截断）
train.group_size每条任务采集的轨迹数（同 algorithm.group_size）
eval.use_fixed_reset_state_ids评估时用固定初始状态（确保可重复）

9. 快速上手

场景一：数学推理 GRPO 训练（Qwen2.5）

# 1. 安装
bash requirements/install.sh

# 2. 启动 Ray 集群（多节点时）
ray start --head --port=6379
# 其他节点：
ray start --address='head_ip:6379'

# 3. 运行训练
bash examples/reasoning/run_main_grpo_math.sh \
  qwen2.5-1.5b-grpo-megatron

# 或直接指定配置文件
python examples/reasoning/main_grpo.py \
  --config-path examples/reasoning/config/math/ \
  --config-name qwen2.5-1.5b-grpo-megatron \
  actor.model.model_path=/path/to/model \
  rollout.model.model_path=/path/to/model
关键配置：该场景 critic.use_critic_model=false、 algorithm.adv_type=grpo，使用 Megatron 训练后端 + vLLM 推理，16 节点。

cluster.num_nodescluster.num_nodes=4节点数量
algorithm.group_sizealgorithm.group_size=16每 prompt 生成样本数
algorithm.kl_betaalgorithm.kl_beta=0.001KL 惩罚（0=不加）
actor.optim.lractor.optim.lr=5e-6学习率
rollout.generation_backendrollout.generation_backend=vllm推理后端
runner.save_intervalrunner.save_interval=50保存间隔（步数）
runner.logger.logger_backends"[wandb]"日志后端
critic.use_critic_modelcritic.use_critic_model=true开启 PPO Critic
algorithm.recompute_logprobsalgorithm.recompute_logprobs=true重计算 logprobs
常见陷阱： 使用 PPO（critic.use_critic_model=true）时，必须同时设置 algorithm.recompute_logprobs=true，否则 Critic 的 value 和 Actor 的 logprobs 会不对齐， 导致 GAE advantages 计算错误。
